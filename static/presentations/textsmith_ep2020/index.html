The original MUD 1

PLAY VIDEO

But that was a single-player text based virtual world. Soon after the appearance of such games some university students in the UK invented MUD.

A MUD (Multi-User-Dungeon) is a multi-player text based adventure game. They were, and continue to be, very very popular in certain parts of the net.

This is a video of me playing the origial MUD yesterday. In it you'll have seen I've just finished chatting with a Wizard called Pyewacket, and come acropper with some rats. It is:

  • Low-tech - telnet (proper clients are available)
  • Quirky - humour, imagination, playfulness and sometimes serious.
  • Literary - narrative driven, sense of place, time and genre (usually fantasy or sci-fi).
  • Lots and lots of fun

There are also many varieties of MUD. In the mid-1990s, while at university I would play on MOO - Multi-user-dungeon Object Oriented worlds.

In a MOO the textual world is a collection of user generated objects. Users program the behaviour of the world as well as inhabit it.

A Rape in Cyberspace

MUD/MOO became very popular. Before all the toxic aspects of current social media platforms, such behaviours were manifested in these virtual textual worlds. Yet at the same time, these were progressive places and I made many friends all over the world at a time when such things were still viewed with incredulity.

Social scientists, psychologists, literary critics, artists and educators all got interested. MIT Professor Sherry Turkle famously analysed such online role playing as a way for folks to explore different aspects of their identity in a relatively safe manner. Challenges that arise allow folks to reflect upon their behaviour and to know more of themselves. A sort of narrative development of their lives.

Being British I spent most of my time asking my fellow players what the weather was like where they were (San Francisco, Paris, Tel Aviv, Singapore). This was strangely satisfying and it made me feel like a true citizen of the global village delivered via the information superhighway that was my 56k modem.

I also built a textual interactive representation of a remote Scottish island my girlfriend (now wife) and I visited over the summer of 1995. I still have a backup, and it's an interactive digital momento from my 21 yo self.

It was, and continues to be, fun to write, explore and interact within such worlds.

What would a MUD/MOO written in 2020 look like?

This all feels very low-tech, but I like that. I don't miss the transactional and manipulative nature of the current state of the art, and the focus is definitely on being in a specific place, at a certain moment, with particular people.

Things I love about MUDs are that they:

  • Tell, rather than show. The user's imagination is paramount. And this is reflected in the simple and uncluttered user interfaces where the text is most important.
  • They can be playful or serious but always tell a story within a recognisable literary genre that sets the scene for interactions between users.
  • Graphical games focus on hand/eye coordination, text based worlds put narrative, meaning and intent at the centre of their process.
  • While the reader may have to do more, the reward is far greater.
  • Literary worlds are more egalitarian and accessible - within the realm of a single developer.
  • Very low overhead in technical terms.
  • Difference between making a film or writing a novel.

So, how does the revised TextSmith add up..?

What parts become the sum total of TextSmith?

  • Quart - Asyncio web framework a la Flask. Provides websocket support. Phil Jones, does an amazing job, and it has been a joy to use.
  • Output to the user is framed within Jinja templates.
  • Babel - slots into Quart. TextSmith is fully translateable. Text based virtual worlds are not just English virtual worlds. I want my code to respect and acknowledge the diversity of folks who may use it.
  • The middleware of the app is Python asyncio and I use some really wonderful libraries to help make this easy to code. Structlog is really helpful for letting me see what the application is doing, and the asyncio related libraries have "just worked" for me.
  • Dave Beazley's SLY (Sly Lex Yacc) is used to process the nascent LISP-y scripting language used by players to create the behaviour of the textual world. Dave's work has been a fun way for me to learn about language design and implementation. Still lots of work to do.
  • Finally, Redis is used as an object store (where Python dictionaries become JSON objects stored against a key). Redis is a fantastic key/value datastore with a host of simple yet powerful features. It also comes with a message queue used by TextSmith for inter-object communication.
  • For example, if a user (an object) says something in a room (another object), then a message from the user, describing what's been said, is sent to the room via the message queue. Then the room itself will send messages to objects representing the other users in the room, telling them someone has said something. It is these messages that are sent to the user's browser via websockets.

TextSmith is also designed to be scalable.

  • Python instances of the app can scale horizontally.
  • All interactions between instances are via the Redis message queue.
  • There is, of course, a Redis cluster sitting behind all this.

However, you could always just run it as a single app with Redis on the same machine.

Steps for Parsing User Input:

  • Escape any HTML in the user input. ;-)
  • Handle special shortcuts (", !, :, @).
  • Check if first word is a built-in verb (get, drop, build etc).
  • Break sentence into: verb, direct-object, preposition, indirect-object.
  • Match verb to an attribute name of objects in the room: user, room, direct object, indirect object.
  • Place the attribute value in the necessary message queues. If the value is a string starting with #!, evaluate as a script.

Script Evaluation (work in progress):

  • Scripts are evaluated in a context containing: user, room, exits, other users, things, this, direct_object, preposition, indirect object, raw user input.
  • Scripts must complete within a certain period of time.
  • Scripts may only update objects within the context.
  • Scripts execute with the privileges of the user who caused them to run.
  • Scripts may only emit messages to objects in the context.
  • If the script fails, users get a meaningful error message / stack trace.
